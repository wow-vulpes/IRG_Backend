# Кейс №2 - IRG: Backend (Java, Spring Boot)

Сервис для управления подписками и регулярными платежами. Реализован на Java 21 + Spring Boot 3.3 в строгом соответствии с ТЗ.

## Стек технологий

- **Язык:** Java 21
- **Фреймворк:** Spring Boot 3.3
- **База данных и миграции:** PostgreSQL, Flyway
- **Сборка проекта:** Maven
- **Тестирование:** JUnit 5, Mockito
- **Инфраструктура:** Docker, Docker Compose
- **API Документация:** Swagger (Springdoc OpenAPI)

## Запуск проекта

1. Чтобы собрать и запустить приложение введите команду:
   ```bash
   docker compose up
   ```
   
2. При старте контейнера автоматически запустится Flyway (инструмент для управления миграциями), который создаст таблицы в базе данных. Приложение будет доступно по адресу `http://localhost:8080`.

3. **Swagger UI (OpenAPI)** доступен по ссылке: `http://localhost:8080/docs`.
Через него можно визуально тестировать все эндпоинты.

## Запуск тестов

Бизнес-логика покрыта unit-тестами (JUnit 5 + Mockito). Покрыты все краевые случаи из ТЗ.

Тесты запускаются **автоматически** при каждой сборке Docker-образа (во время шага `RUN mvn clean package` в Dockerfile). 
Если вы хотите запустить их вручную **вне Докера** (локально), убедитесь, что у вас установлены Java 21 и Maven, и выполните команду в корневой директории:
```bash
mvn test
```
*(Примечание: если локально Maven не установлен, опирайтесь на автоматический прогон тестов внутри Docker при сборке).*

## Ответы на вопросы из ТЗ

- **Почему lazy expiry не применяется к рекуррентным обязательствам.**
   
   Жизненный цикл рекуррентной подписки не заканчивается в дату очередного платежа. Если дата наступила или прошла, подписка должна оставаться в статусе `ACTIVE`, чтобы пользователь мог её оплатить (сдвинув дату на следующий период) или отменить вручную. Разовые же платежи (`recurrence IS NULL`) логически теряют актуальность после своей даты, поэтому только они автоматически переводятся в статус `EXPIRED`.

- **Как обрабатываются граничные случаи с датами при рекуррентности.**

   Например, если у платежа дата 31 января, то при очередном списании дата должна стать 28 февраля (в високосный год - 29), а не 2 марта.

   Используется встроенный в Java механизм `LocalDate.plusMonths(1)`. Он корректно обрезает дни для месяцев с меньшим количеством дней (включая високосные феврали). Например, `LocalDate.of(2023, 1, 31).plusMonths(1)` даст ровно `28 февраля`. Ошибка смещения не накапливается, потому что логика опирается на строгую календарную математику, а не просто прибавляет 30 дней.

- **Какие компромиссы сделал и что бы сделал иначе при наличии большего времени.**

   Добавила бы авторизацию (Security).

   Ради фокуса на бизнес-логике ТЗ, слой Spring Security был опущен. В реальном проекте каждый эндпоинт требовал бы привязки к `user_id` из JWT-токена.

## Примеры запросов (cURL)

**Создание подписки:**
```bash
curl -X 'POST' 'http://localhost:8080/obligations' \
  -H 'Content-Type: application/json' \
  -d '{
  "title": "Netflix",
  "amount": 299.50,
  "currency": "RUB",
  "category": "SUBSCRIPTION",
  "recurrence": "MONTHLY",
  "next_payment_date": "2026-07-18"
}'
```

**Оплата обязательства (нужно подставить UUID):**
```bash
curl -X 'POST' 'http://localhost:8080/obligations/e2e01901-a077-4128-b5fa-b796787d939d/pay'
```

**Получение списка всех обязательств (с фильтрацией):**
```bash
curl -X 'GET' 'http://localhost:8080/obligations?category=SUBSCRIPTION&status=ACTIVE'
```

**Получение списка ближайших платежей (по умолчанию за 7 дней):**
```bash
curl -X 'GET' 'http://localhost:8080/obligations/upcoming?days=14'
```

**Отмена обязательства:**
```bash
curl -X 'PATCH' 'http://localhost:8080/obligations/e2e01901-a077-4128-b5fa-b796787d939d/cancel'
```

**Удаление обязательства (также триггерит SSE событие):**
```bash
curl -X 'DELETE' 'http://localhost:8080/obligations/e2e01901-a077-4128-b5fa-b796787d939d'
```
