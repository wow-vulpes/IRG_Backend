# Subscription Registry API

Сервис для управления подписками и регулярными платежами. Реализован на Java 21 + Spring Boot 3.3 в строгом соответствии с ТЗ.

## Запуск проекта

1. Запустите инфраструктуру (PostgreSQL) через Docker (настроен в `docker-compose.yml` в корне):
   ```bash
   docker-compose up -d
   ```
2. Соберите и запустите приложение (база данных будет автоматически проинициализирована и мигрирована через Flyway):
   ```bash
   mvn clean install
   mvn spring-boot:run
   ```

## Архитектурные решения

- **Lazy Expiry (Ленивое истечение сроков)**: Вместо того чтобы каждую минуту сканировать базу cron-джобом (что создало бы огромную паразитную нагрузку на БД), мы переводим просроченные платежи в статус `EXPIRED` на лету (just-in-time). Это происходит перед формированием списков (`GET /obligations` и `GET /upcoming`) и при точечном обращении к записи (перед оплатой или отменой).
- **Server-Sent Events (SSE)**: Для мгновенного оповещения фронтенда используется `SseEmitter` (`GET /obligations/stream`). При вызове `DELETE /obligations/{id}` сервер "пушит" однонаправленное событие клиентам. Это проще и легковеснее, чем WebSockets.
- **KISS & Прагматичность**: Использованы прямые классы-сервисы без избыточных интерфейсов (спасибо CGLIB). Обработка ошибок вынесена в единый `GlobalExceptionHandler`.

## Ответы на вопросы из ТЗ и краевые случаи

1. **Сдвиг даты (31 января + 1 месяц)**:
   При рекуррентных платежах используется стандартный механизм `java.time.LocalDate`. Метод `plusMonths(1)` идеально решает эту задачу "из коробки", корректно обрезая дни в феврале (и учитывая високосные годы). Например, `LocalDate.of(2023, 1, 31).plusMonths(1)` возвращает ровно `28 февраля`. Дальнейшие сдвиги не накапливают ошибку смещения, так как привязаны к изначальному графику.
   
2. **Тип данных `DECIMAL(15, 2)` для денег**:
   Хранить финансовые данные в `FLOAT` или `DOUBLE` недопустимо из-за фундаментальной потери точности в стандарте IEEE 754 (проблема вида `0.1 + 0.2 = 0.30000000000000004`). В Java для денег используется `BigDecimal`, а в PostgreSQL — `DECIMAL(precision, scale)`. `15, 2` означает, что максимальная длина числа составляет 15 знаков, из которых 2 строго зарезервированы под копейки/центы.
